# Standard Workflow Skill Design

## Context

North Star module: Workflow Runtime Specification and Workflow Toolkit.

The strategy describes a Workflow Toolkit with at least Standard, Lightweight, and Bugfix workflows. Current generated Harness assets only include:

- `.ai/skills/lightweight/SKILL.md`
- `.ai/skills/bugfix/SKILL.md`

`harness-config.yaml` also only references `lightweight` and `bugfix`. This leaves a gap for complex, cross-module, high-risk, security, data migration, or architecture-impacting tasks. Those tasks should not be routed through lightweight by default.

## Current Gap

Harness Builder has no generated Standard Workflow Skill and no config entry that a host AI Coding Runtime can select for high-risk work.

The gap is not runtime execution. Harness Builder still does not run task workflows or generate `.ai/task-runs`. The gap is a missing static Workflow Toolkit asset and config contract.

## Decision

Add a fixed-template Standard Workflow Skill:

- `.ai/skills/standard/SKILL.md`
- `harness-config.yaml` workflow entry: `standard`

The Standard Skill will define:

- usage conditions for complex/high-risk/cross-module work;
- required guides and sensors;
- stages: requirement alignment, harness mapping, solution design, implementation plan, test-first build and verify loop, review/handoff;
- runtime artifact contract owned by host AI Coding Runtime;
- review-only Experience Candidate handling.

This remains deterministic template generation. No LLM-generated workflow skill content is introduced.

## Boundaries

- Do not restore or depend on `harness-builder-agent run`.
- Do not generate `.ai/task-runs`.
- Do not dynamically generate Workflow Skills with LLM.
- Do not make Standard the default workflow yet; default remains `lightweight`.
- Do not change Harness Map runtime schema beyond allowing `standard` as a future selected workflow contract.

## Acceptance Criteria

- `write_skill_assets` writes `.ai/skills/standard/SKILL.md`.
- `HarnessConfig.default()` contains `standard` with `skill_path: .ai/skills/standard/SKILL.md`.
- `init` integration fixtures generate the Standard Skill and config reference.
- Benchmark requires the Standard Skill file and content in `content:workflow-skills`.
- Missing Standard Skill causes workflow skill benchmark content check to fail.
- Harness Map schema accepts `standard` as a future runtime selected workflow.
- Tests prove Runtime task-run artifacts are still only contracts and are not generated by Builder.

## Assumptions

- Standard Workflow is the safest next Toolkit increment because it is named in the strategy and needed before more dynamic workflow selection.
- Prototype-first workflow and migration/test-generation workflows should be later separate milestones.
- The default workflow should stay `lightweight` to avoid changing existing low-risk behavior.

## Risks

- Adding a workflow config entry without dynamic routing may look static. That is acceptable for this milestone because current architecture explicitly uses fixed templates and avoids dynamic LLM workflow generation.
- Benchmark expectations must be updated carefully so all generated fixtures include the new skill before checks require it.
