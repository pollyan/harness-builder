# Maturity Evidence Pack Design

## Context

This is the second autonomous milestone in the strategy-guided loop.

North Star reference:

- `docs/strategy/Harness Builder — 面向遗留代码库治理的 AI Coding Harness 生成器.md`
- `4.1 Core Harness 与 Improvement System`
- `6.6 Maturity & Evolution：成熟度评估与演进路径`

Previous milestone:

- `Maturity Model v2` added structured maturity dimensions, blockers, caps, and next steps.

## Current State Gap Analysis

The strategy defines Maturity & Evolution as a roadmap engine that should combine Scanner, Experience, Maturity, and Recommendation. The new `maturity-score.yaml` now records the roadmap, but there is still no single auditable evidence input that explains what the maturity evaluator saw.

Current evidence is scattered across:

- `.ai/project-inventory.json`
- `.ai/command-catalog.yaml`
- `.ai/harness-config.yaml`
- `.ai/scan-metadata.yaml`
- `.ai/llm-scan-proposal.json`
- `.ai/weapon-library-selection.yaml`
- `.ai/guides/`
- `.ai/sensors/`
- `.ai/skills/`
- `.ai/runs/`
- `.ai/experience/pending-improvements.md`
- `.ai/benchmark-report.yaml`

This blocks later `LLM Maturity Reviewer` work because the LLM should not freely inspect arbitrary repo state and infer from ad hoc context. It needs a deterministic, schema-validated evidence pack first.

## Chosen Milestone

Implement **Maturity Evidence Pack**.

The milestone adds a machine-readable `.ai/maturity-evidence.yaml` generated by `init` and refreshed by `assess`. It summarizes the harness assets, commands, generated traces, experience files, benchmark status, and known runtime artifact availability.

## Alternatives Considered

### Option A: Let the future LLM reviewer read all `.ai` files directly

Rejected. It increases prompt drift and makes evidence harder to test.

### Option B: Skip evidence pack and make `improve` consume `maturity-score.yaml`

Rejected as the next milestone because `improve` would still lack source context for rationale and acceptance checks.

### Option C: Add deterministic evidence pack first

Selected. It is small, testable, and directly supports LLM maturity review and maturity-driven improve.

## Product Decisions

1. The evidence pack is deterministic and schema-validated.
2. It does not call LLM.
3. It does not change formal Harness assets by itself.
4. It treats `.ai/task-runs` as optional external Runtime evidence and only records whether it is present.
5. It is generated during `init` and refreshed during `assess`.
6. It is included in benchmark schema checks so regressions are caught.

## Data Contract

Add `src/harness_builder_agent/schemas/maturity_evidence.py`.

Schema:

```python
MaturityEvidencePack:
  schema_version: str = "1.0"
  repo_name: str
  primary_stack: str
  generated_at: str | None
  inventory_summary: InventoryEvidenceSummary
  command_summary: CommandEvidenceSummary
  harness_assets: HarnessAssetEvidence
  observability: ObservabilityEvidence
  experience: ExperienceEvidence
  benchmark: BenchmarkEvidence | None
  maturity_inputs: list[str]
  warnings: list[str]
```

The first implementation keeps fields summary-level. It does not duplicate full source documents or generated Markdown.

## Output

New machine asset:

```text
.ai/maturity-evidence.yaml
```

It is a current-state evidence pack for:

- deterministic maturity assessment;
- future LLM maturity review;
- maturity-driven improve candidate rationale.

## Implementation Plan Shape

New tool:

```text
src/harness_builder_agent/tools/maturity_evidence.py
```

Responsibilities:

- read current `.ai` assets;
- count guides/sensors/skills;
- summarize command catalog gate distribution;
- summarize generation trace availability;
- detect pending improvements;
- detect optional runtime task-runs;
- detect benchmark report if present;
- return `MaturityEvidencePack`.

Writers:

- `write_report_assets()` writes initial evidence pack using current in-memory inventory, commands, config, and weapon selection.
- `assess_maturity()` refreshes evidence pack from disk after reading current `.ai` assets.

Benchmark:

- Add `schema:maturity-evidence` check.

## Assumptions

- `maturity-evidence.yaml` can be generated before `skills` are physically written during `init`; in that path it may use planned config/weapon selection as evidence and later `assess` refreshes from disk.
- Runtime task-run evidence is optional and absent in current Harness Builder CLI runs.
- Existing acceptance does not need real DeepSeek changes for this milestone because evidence pack is deterministic.

## Open Risks

- The pack can grow too broad if it starts copying file contents. This milestone only records summaries and paths.
- Later LLM reviewer may require more detailed evidence slices; that should be a future targeted extension.

## Acceptance Criteria

- `.ai/maturity-evidence.yaml` validates against `MaturityEvidencePack`.
- `init` emits the file and records it in generation trace artifacts.
- `assess` refreshes the file.
- benchmark validates `schema:maturity-evidence`.
- Fast regression passes.
