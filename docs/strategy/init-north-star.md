# Init North Star：深度引导式 Harness 生成体验

本文定义 `harness-builder-agent init` 的目标态、用户体验原则、质量指标和后续迭代边界。它用于约束后续目标模式：短期内所有主要开发循环都应优先服务 `init` 主线，避免在候选治理、Runtime、benchmark、prompt 等周边能力之间继续发散。

## 核心定位

`init` 不是模板初始化命令，而是 Harness Builder 面向用户的第一性体验。

用户拿一个陌生或遗留代码库运行 `init` 后，Harness Builder 应该像一个资深工程顾问一样：

1. 深入理解仓库结构、技术栈、模块边界、验证能力和风险区域。
2. 基于成熟度框架评价当前 Harness 现状，说明当前处于哪个阶段、为什么、距离下一阶段缺什么。
3. 清楚解释它看到了什么、为什么这样判断、哪些地方不确定。
4. 让用户以低成本确认、修正或补充关键上下文。
5. 围绕成熟度缺口推荐针对当前仓库的 Guides、Sensors、Workflow Skills、配置和后续改进动作。
6. 给出可审计的证据链、下一步建议和质量门禁解释。
7. 让用户感觉这是“为这个仓库定制的 Harness”，而不是固定模板拼装。

## 目标用户场景

### 首次初始化

作为 Harness Maintainer，我希望对一个已有代码库运行 `init` 后，系统能理解仓库、解释判断、引导我补充关键上下文，并生成一套可审查、可编辑、可持续演进的 `.ai/` Harness。

成功体验应该是：

- 我知道系统为什么判断这是某个技术栈。
- 我知道系统发现了哪些模块、验证命令、风险和缺口。
- 我知道当前仓库的 AI Coding Harness 成熟度处于 L0-L4 中哪个阶段。
- 我知道当前等级的证据是什么、下一等级缺什么。
- 我知道哪些内容是事实、哪些是推断、哪些需要人工确认。
- 我能快速看到生成了哪些资产、它们之间如何关联。
- 我知道下一步应该先看哪些文件、先跑什么命令、如何判断质量。

### 再次进入已有 Harness

作为 Harness Maintainer，我希望再次运行 `init` 时，系统不是重新覆盖资产，而是先展示已有 Harness 的健康状态、维护建议和下一步动作。

成功体验应该是：

- 我能看到成熟度、benchmark、Experience、review-only 候选、Runtime 过程数据等关键信号。
- 我能理解最应该先处理的问题是什么。
- 我可以选择 assess、improve、benchmark、recommend-workflow、review-candidate、self-improve 或 reinit。
- 系统不会在我没有明确选择前覆盖正式 Guides、Sensors、Workflow Skills 或配置。

## 目标态 CLI 交互旅程

`init` 的目标交互形态是 CLI 中的渐进式协作式 Agent 工作流，而不是先一次性完成扫描与生成，再用 Markdown 文件让用户事后补充。

Markdown 产物是审计、交付和后续维护材料；它们不是 `init` 过程中与用户对齐信息的主要界面。`init` 的主要交互界面是 CLI。

渐进式协作是硬约束：系统不能先在后台把扫描、判断、Guide / Sensor / Workflow 设计全部做完，再在最后装作和用户确认。每个关键阶段都应在“拿到一批信息 -> 展示理解 -> 请用户确认或补充 -> 吸收补充 -> 再进入下一阶段”的循环中推进。

### 1. 启动与目标说明

用户运行 `harness-builder-agent init --repo <repo>` 后，CLI 应先用简短清晰的方式说明本次流程：

- 将扫描哪些信息。
- 需要用户确认哪些关键判断。
- 会生成哪些 `.ai/` Harness 资产。
- 哪些动作不会发生，例如不会执行 Runtime、不创建 `.ai/task-runs`、不会在未确认时覆盖已有 Harness。

目标体验：用户在等待扫描前就知道流程边界和预期产出。

### 2. 阶段化扫描与进度反馈

扫描阶段不能长时间静默。

CLI 应按阶段输出进度，例如：

- 正在收集仓库文件和配置证据。
- 正在识别构建、测试、CI 和运行命令。
- 正在抽取源码入口、模块线索和风险区域。
- 正在请求 LLM 进行结构化分析。
- 正在调和 LLM 判断与确定性 evidence。

每个长时间阶段都应有状态反馈。即使无法显示精确百分比，也应至少显示当前阶段、已完成阶段和下一阶段，让用户不会误以为程序卡死。

目标体验：用户知道系统正在工作，并能理解耗时来自哪里。

### 3. 扫描结果的友好呈现

扫描和初步分析完成后，CLI 应暂停进入“对齐扫描理解”阶段，而不是立刻写入最终资产。

CLI 应用清晰排版展示：

- 技术栈判断：primary stack、候选 stack、置信度、证据路径。
- 模块理解：主要目录、模块职责、入口文件、测试目录。
- 验证能力：发现的 build / test / lint / run / CI 命令，gate 类型和置信度。
- 风险和不确定性：低置信度判断、缺失测试、命令不可验证、模块边界不清、潜在高风险目录。
- 系统仍需要用户补充的问题。

这些内容应通过阶段标题、分组、缩进、颜色或强调形成视觉焦点。颜色只能辅助，不能成为唯一信息载体。

目标体验：用户能快速判断“系统是否理解了我的仓库”，并知道应该纠正哪里。

### 4. 成熟度初评与差距解释

扫描理解展示之后，CLI 应立即把这些信息放入成熟度框架中解释，而不是只停留在“扫描到了什么”。

目标态展示应包括：

- 当前 Harness 成熟度等级：L0-L4。如果仓库没有任何 `.ai/` Harness 或 AI Coding 控制资产，应明确视为从 L0 起步。
- 等级解释：当前等级对用户意味着什么，能支撑什么 AI Coding 控制能力，不能支撑什么能力。
- 下一等级差距：从当前等级到目标下一等级需要补齐什么能力。
- 评分证据：当前等级判断对应哪些文件、命令、目录、CI 配置、测试能力、风险线索或缺失项。
- 阻断原因：为什么当前还不能达到下一等级，例如缺少可执行 Sensor、风险区域未确认、Workflow routing 未被 Runtime 证据验证。
- 推荐补齐动作：哪些 Guides、Sensors、Workflow routing、人工确认项或 benchmark 应优先生成。

成熟度解释必须先讲 L0-L4 等级，再用维度评分解释为什么是这个等级。机器字段可以落到 YAML / JSON，CLI 中应把它翻译成“当前等级、证据、阻断、下一步”。

目标体验：用户不是只知道“扫描结果”，而是知道“当前 AI Coding Harness 能力好不好、为什么、下一步怎么变好”。

### 5. 与用户对齐扫描理解和成熟度判断

CLI 应在展示扫描结果后主动询问用户是否需要补充或修正，而不是只接受固定格式命令。

目标态应支持两类输入：

- 低门槛自然语言补充，例如“后端主要在 services/order，测试要用 make test-integration，支付模块风险最高”。
- 高精度结构化修正，例如 stack、module、command、risk 的明确字段。

系统收到补充后，应重新整理用户输入，明确展示“我将如何理解这些补充”，并说明这些补充如何影响成熟度评分、缺口判断和 Harness 推荐。重要补充必须进入 machine-readable decisions 或 context inputs，不能只停留在屏幕输出。

目标体验：用户不需要学习内部格式，也能让系统理解业务上下文、团队规则和真实验证方式。

### 6. 渐进式深入，而不是一次性生成

`init` 应逐步发掘信息、逐步对齐信息、再生成 Harness。

目标态流程应是：

1. 先收集基础 evidence。
2. LLM 提出初步结构化理解。
3. CLI 展示扫描理解。
4. 基于成熟度框架给出初评和差距。
5. 请求用户确认或补充关键判断。
6. 根据用户补充、不确定点和成熟度缺口，必要时继续补充 evidence 或重新调和判断。
7. 再进入 Guide / Sensor / Workflow 设计。
8. 对生成方案做 maturity-driven preview。
9. 用户确认后写入 `.ai/` 资产。

这意味着后续 Guide、Sensor 和 Workflow Skill 生成必须消费用户在前序交互中确认或补充的信息，并围绕成熟度缺口组织。不能在用户输入前就把所有设计一次性定死。

每一轮协作都必须留下可消费结果：

- 用户确认或修正的扫描理解，应进入后续成熟度诊断。
- 用户补充的业务风险、团队规则和验证命令，应进入后续 Guide / Sensor / Workflow 推荐。
- 用户跳过的问题，应进入不确定性和 `human-input-needed`。
- 系统基于用户输入改变了哪些判断，应在 CLI 中复述，并在 trace / decisions 中可审计。

反模式：

- 先完整生成 `.ai/`，最后询问用户“是否满意”。
- 用户补充只影响屏幕文字，不影响后续生成。
- 把互动做成固定问卷，而不是基于当前仓库证据和成熟度缺口动态追问。
- 为了显得智能而重复询问已经有明确证据支持的问题。

目标体验：用户感到自己是在和一个 Agent 一起逐步理解仓库，而不是回答一个事后问卷。

### 7. 成熟度驱动的 Harness 设计预览

在正式写入 `.ai/` 前，CLI 应展示 Harness 设计预览：

- 将生成哪些 Guides，每个 Guide 为什么需要。
- 将生成哪些 Sensors，每个 Sensor 对应哪些验证命令或缺口。
- 将生成哪些 Workflow Skills，以及它们如何引用 Guides / Sensors。
- routing policy 如何区分 bugfix、lightweight、standard。
- 哪些内容标记为低置信度或需要人工确认。
- 每个推荐项对应哪个成熟度维度、解决哪个阻断项、预计帮助进入哪个下一阶段能力。

用户应能选择确认、返回修改扫描理解、补充团队规则，或取消。

目标体验：用户在写入前能理解即将得到的 Harness，而不是事后打开一堆文件猜测含义。

### 8. 写入后的交付摘要

写入完成后，CLI 应给出短交付摘要，而不是只打印文件路径。

摘要应包含：

- 生成结果。
- 当前成熟度判断、主要证据和下一等级缺口。
- 最值得先看的 3 到 5 个文件。
- 是否尚未运行 benchmark。
- 建议下一步命令。
- 仍需人工确认的问题。

`init-summary.md` 是该交付摘要的可持久化版本，用于会后审查和团队协作。

目标体验：用户知道下一步该干什么，并能把结果交给团队继续看。

## 体验原则

### 成熟度是叙事主线

`init` 的主线不是“扫描 -> 生成文件”，而是“扫描 -> 成熟度诊断 -> 缺口解释 -> Harness 推荐 -> 用户确认 -> 建立基线”。

所有关键输出都应围绕成熟度组织：

- 扫描结果用于支撑成熟度评分，而不是孤立展示。
- 用户补充用于修正成熟度判断和推荐优先级。
- Guide / Sensor / Workflow 推荐必须说明它补齐哪个成熟度维度。
- `init-summary.md` 应解释当前等级、目标下一等级、阻断原因和推荐行动。
- benchmark 应验证生成资产是否真的支撑这些成熟度主张。

当仓库没有任何现有 Harness 时，系统应明确说明“当前 Harness 成熟度从 L0 起步”，并把第一版 `.ai/` 资产解释为建立 L1/L2 基线的起点，而不是暗示仓库本身已经具备成熟 Harness。

### 成熟度叙事契约

面向用户的成熟度叙事只使用 L0-L4 五个等级作为主线。内部维度评分用于解释等级，不应喧宾夺主。

- L0：没有项目级 AI Coding Harness。仓库可能有普通工程资产，但尚未形成可供 AI Coding 使用的 Guides、Sensors、Workflow 或治理基线。
- L1：有初步 Harness 基线。系统已能生成或发现部分项目上下文、规则、验证命令或风险线索，但覆盖有限，仍依赖人工判断。
- L2：有可用的项目级控制资产。Guides、Sensors、Workflow routing 和成熟度报告基本成形，能支撑常见 AI Coding 任务前的上下文加载和任务后的基础验证。
- L3：有运行证据闭环。Harness 不只存在于文件中，还能被 Runtime / 任务过程数据验证，Sensor 结果、Repair Loop、人工决策和审计记录能反哺成熟度。
- L4：有持续自优化能力。Harness 能基于多次任务趋势、失败模式、Review 反馈和成熟度变化，持续生成可审查的改进候选，并逐步优化规则、Sensors 和 Workflow 策略。

CLI 中应优先回答：

1. 当前是 L几。
2. 为什么是这个等级。
3. 距离下一级差什么。
4. 本次 `init` 推荐生成或补齐什么。
5. 哪些推荐需要用户确认后才能进入正式 Harness。

如果需要展开细节，再进入维度解释，例如 Guides、Sensors、Workflow、Risk Control、Observability、Experience、Governance 等。维度解释必须服务于 L0-L4 主线。

### 深度优先

`init` 必须优先提升仓库理解深度，而不是只追求产物数量。

后续迭代应优先强化：

- evidence collection：构建配置、CI、源码入口、测试入口、文档、配置、关键模块。
- LLM-guided evidence expansion：LLM 可以提出还需要看的文件，Python 负责 allowlist、读取和校验。
- stack / module / command / risk 的证据链与置信度。
- 企业不规范仓库中的非标准目录、缺失测试、混合技术栈和隐性风险。
- 围绕 Harness 推荐反推所需信息：为了推荐 Guides、Sensors、Workflow routing 和风险策略，必须提前收集足够的项目结构、验证能力、团队规范、业务风险、数据/权限/迁移风险和现有工程控制证据。

### 可解释

用户不能只看到结论，必须看到判断依据。

`init` 输出和 `init-summary.md` 应稳定表达：

- 发现了什么。
- 证据来自哪里。
- 为什么做出这个判断。
- 哪些地方置信度低。
- 哪些事项需要人工确认。
- 这些判断如何影响生成的 Guides、Sensors 和 Workflow Skills。
- 这些判断如何影响成熟度等级、阻断项和下一步推荐。

CLI 面向用户的内容必须优先是人能读懂的叙述，而不是内部字段或代码化对象的直接展开。机器消费的 JSON / YAML 应保留完整结构；面向用户的 CLI 和 Markdown 应把结构化数据翻译成业务含义、工程影响和下一步动作。

### 交互低负担

guided `init` 应降低用户输入成本。

目标态不是要求用户记住 `module=路径|类型|名称` 这类格式，而是逐步走向：

- 先展示系统判断，再询问是否修正。
- 支持自然语言补充团队规则、架构约束、测试策略和业务风险。
- 对结构化修正提供清晰示例和容错提示。
- 每一步都说明为什么需要这类信息。
- 用户补充后，系统应复述自己的理解并把补充纳入后续生成。
- 用户可以跳过非关键问题，但跳过后的不确定性必须记录到产物。

交互必须发生在决策形成过程中，而不是产物生成之后。凡是会影响成熟度判断、Guide / Sensor 内容、Workflow routing 或风险策略的用户输入，都必须在对应设计决策完成前收集，并被后续步骤消费。

### 视觉焦点清晰

CLI 是当前主要 UI，因此终端体验必须被当作产品界面设计。

目标态：

- 使用清晰的阶段标题，让用户知道当前处于扫描、证据扩展、LLM 分析、调和、资产生成、总结中的哪一步。
- 使用颜色或强调区分成功、警告、失败、需要人工确认和下一步动作。
- 使用一致的缩进、列表、分组和摘要格式，避免大段无层次文本。
- 长时间任务必须有进度反馈或阶段状态，不能让终端长时间无输出。
- 颜色必须可关闭或在无颜色环境下仍可读，不能只靠颜色表达关键信息。
- 错误信息应包含原因、影响和下一步处理建议。

### 可审计

`init` 的每个关键判断都应能回溯。

必须保留：

- LLM 原始结构化 proposal。
- reconciled inventory。
- command catalog。
- interaction decisions。
- generation trace。
- artifacts list。
- human-input-needed。
- maturity evidence。

LLM 输出不能直接成为不可审计事实；需要 schema、evidence source、allowlist、reconciler 和 benchmark 共同约束。

## 渐进式交互状态机

`init` 的渐进式协作不应只停留在流程描述中。每个关键阶段都必须形成“收集信息、展示理解、吸收用户输入、影响后续决策、留下机器可读记录”的闭环。

| 阶段 | 系统已收集信息 | CLI 应展示 | 用户可补充 | 后续受影响决策 | 机器可读记录 |
|---|---|---|---|---|---|
| 启动说明 | repo 路径、目标输出目录、已有 `.ai/` 状态 | 本次流程边界、不会执行的动作、预计产物 | 是否继续、是否进入维护入口 | 首次 init 或已有 Harness 维护路径 | init session metadata |
| 基础扫描 | 文件树、构建配置、CI、测试目录、文档、入口文件 | 当前扫描阶段、发现的关键证据、不确定项 | 非标准目录、真实测试入口、团队约定位置 | 是否需要扩展 evidence、哪些判断低置信度 | project inventory、command catalog |
| LLM 结构化理解 | evidence 摘要、候选 stack/module/command/risk | 技术栈、模块、命令、风险的初步判断和证据 | 修正 stack、模块边界、风险区域、验证命令 | reconciled inventory、后续 maturity 输入 | LLM proposal、reconciled inventory |
| 成熟度初评 | reconciled inventory、已有 Harness 资产、缺失项 | 当前 L0-L4、为什么、下一等级差距、阻断项 | 团队规则、业务风险、控制目标、可接受严格度 | maturity level、gap、推荐优先级 | maturity evidence、context inputs |
| 深度追问 | 低置信度点、成熟度缺口、风险高影响点 | 需要补充的问题及原因 | 自然语言或结构化补充，也可以跳过 | Guide / Sensor / Workflow 设计是否可信 | interaction decisions、human-input-needed |
| 设计预览 | 扫描结论、用户补充、成熟度缺口 | 将生成的 Guides、Sensors、Workflow、routing 和预期提升 | 确认、修正、取消、补充团队规则 | 写入正式资产或候选资产 | generation trace、artifacts plan |
| 写入总结 | 写入结果、剩余不确定项、下一步建议 | 生成文件、当前 L 等级、下一等级缺口、最该看的文件 | 后续命令选择或退出 | assess / benchmark / improve / maintenance 入口 | init-summary、artifacts list |

任何用户输入只要影响成熟度、风险、Guide、Sensor 或 Workflow，就必须进入后续决策链路。只在屏幕上复述但不改变后续生成结果，视为无效交互。

## Harness 推荐信息需求

`init` 的扫描深度应由“为了推荐可靠 Harness 需要知道什么”反推，而不是由固定文件模板反推。以下信息不是静态现状基线，而是目标态推荐所需的输入类型。

| 推荐对象 | 至少需要的输入 | 缺失时的处理 |
|---|---|---|
| Guides | 技术栈、模块职责、关键路径、团队规范、架构边界、核心业务流程、常见风险 | 转成用户确认问题或 `human-input-needed`，不能伪造项目规则 |
| Sensors | 可执行验证命令、CI 线索、测试目录、缺失测试、命令风险、运行前置条件 | 标记 hard / soft / low-confidence gate，并说明不可验证原因 |
| Workflow Skills | 任务类型、风险区域、Sensor 覆盖、人工升级条件、可接受自动化程度 | 选择保守 routing，低置信度部分进入候选说明 |
| 成熟度判断 | 已有 Harness 资产、Guides/Sensors/Workflow 完整性、Runtime 证据、候选改进、审计记录 | 用 L0-L4 解释能力边界，缺少 Runtime 证据时不能假定 L3/L4 |
| 改进建议 | 成熟度缺口、用户补充、风险集中区、验证能力缺口、已有候选状态 | 只生成可审查建议，不自动覆盖正式资产 |

当代码证据不足以支持推荐时，系统应明确说明“目前不知道什么、为什么重要、用户补充后会影响什么”，而不是输出看似完整但没有证据的 Harness。

## 成熟度推荐表达模板

CLI、`init-summary.md` 和生成前 preview 中的推荐应使用一致的成熟度叙事语法：

```text
当前等级：Lx
判断依据：来自哪些扫描证据、已有 Harness 资产或用户补充
下一等级差距：缺少哪类 Guides / Sensors / Workflow / Runtime evidence / governance
推荐动作：本次 init 建议生成、补齐或标记为候选的内容
预期提升：该动作如何帮助进入或接近下一等级
确认状态：已确认 / 需用户确认 / 低置信度候选 / 暂缓
```

推荐项必须先说 L0-L4，再解释维度。维度评分只能回答“为什么是这个等级”和“下一等级缺什么”，不能替代等级叙事。

## CLI 展示规范

CLI 是当前主要产品界面，输出应形成稳定的视觉语言。

- 阶段标题使用一致格式，例如 `== 扫描仓库 ==`、`== 成熟度初评 ==`、`== 生成预览 ==`。
- 状态应区分完成、警告、需要确认、失败和下一步动作；颜色可以辅助，但无颜色环境下仍必须靠文字和符号可读。
- 长任务至少输出当前阶段、已完成阶段和下一阶段；无法给出百分比时，不要伪造精确进度。
- 扫描结果应按 stack、modules、commands、risks、uncertainties 分组，避免直接 dump 内部 JSON / YAML 字段。
- 成熟度区域应先显示 L0-L4 等级、依据、阻断项和下一步，再按需展开维度。
- 错误信息必须包含原因、影响和可执行下一步。
- 写入前 preview 和写入后 summary 应保持相同的命名、排序和术语，降低用户理解成本。

## 北极星验收口径

后续 milestone 不能只用“文件生成成功”证明 `init` 变好。每个 `init` 体验切片至少应选择一组可验证口径：

- CLI transcript 能展示阶段进度、扫描理解、L0-L4 成熟度初评、下一等级差距和写入前 preview。
- 用户补充能被写入 machine-readable decisions 或 context inputs，并改变后续成熟度判断、推荐或资产内容。
- `init-summary.md` 能作为交付摘要阅读，解释当前 L 等级、证据、阻断项、生成内容和下一步。
- Guides / Sensors / Workflow 含有目标仓库特定路径、模块、命令或风险，而不是泛化模板。
- 低置信度、高风险或缺失证据的内容进入候选、待确认或 `human-input-needed`，不伪装成事实。
- benchmark 或测试断言 schema、关键章节、跨文件引用、成熟度叙事和证据链，而不只断言文件存在。

## Init 质量指标

后续 `init` 相关 milestone 应尽量把“体验变好”转成可验证标准。

### 仓库理解指标

一次成功的首次 `init` 至少应该：

- 识别 primary stack，并给出 evidence path 与置信度。
- 识别主要模块或明确说明模块边界不清。
- 识别构建、测试、lint、运行或 CI 相关命令候选。
- 区分 hard gate、soft gate 和低置信度命令。
- 识别至少一类风险区域或说明风险证据不足。
- 记录缺失测试、缺失 CI、命令不可验证等不确定性。
- 识别与 Harness 推荐相关的上下文：团队规则、架构边界、核心业务流程、权限/安全/数据风险、迁移风险、发布约束、验证环境限制；无法从代码中确认时应转成用户确认问题。

### 成熟度诊断指标

一次成功的首次 `init` 至少应该：

- 明确当前 Harness 成熟度等级。如果没有现有 `.ai/` Harness 或等价控制资产，应说明当前从 L0 起步。
- 用 L0-L4 五级语言解释当前等级、当前等级能力边界和下一等级差距。
- 在需要展开时展示关键维度评分，包括 Guides、Sensors、Workflow、Risk Control、Repair Loop、Observability、Experience、Verification Sophistication、Governance / Auditability。
- 对影响下一等级的低分维度给出证据、阻断原因和下一等级要求。
- 把推荐的 Guide / Sensor / Workflow / human confirmation 与具体成熟度缺口关联起来。
- 说明哪些成熟度判断来自扫描证据，哪些来自用户补充，哪些仍是低置信度推断。
- 用面向人的语言解释评分，不直接把内部 schema 字段当作用户界面。

### 资产质量指标

生成的 `.ai/` 资产应该：

- Guides 包含真实仓库路径、模块名、技术栈和验证命令。
- Sensors 包含当前可运行的验证命令、缺失验证能力和失败处理策略。
- Workflow Skills 引用真实存在的 Guides / Sensors。
- `harness-config.yaml` 的 routing policy 能解释何时选择 bugfix、lightweight 或 standard。
- `init-summary.md` 像交付报告，而不是文件清单。
- `human-input-needed.md` 明确列出后续需要人工确认的问题。
- 每个主要资产都能解释它服务的成熟度维度和补齐的能力缺口。

### 体验质量指标

guided `init` 应该：

- 在长时间 LLM / scan 阶段持续输出阶段状态。
- 在每个阶段结束时给出简短摘要。
- 在扫描后展示成熟度初评和下一等级差距。
- 对用户输入给出清晰提示、默认选项和错误恢复。
- 在用户补充后复述理解，并说明对成熟度判断或 Harness 推荐的影响。
- 在写入前展示将生成或更新的资产摘要。
- 对已有 Harness 默认进入维护入口，而不是覆盖。

### Benchmark 指标

benchmark 不应只检查文件存在。

与 `init` 相关的 benchmark 应逐步覆盖：

- schema 有效性。
- 关键章节存在。
- 关键字段与跨文件引用一致。
- Guide / Sensor 是否包含仓库特定内容。
- hard gate command 是否有 evidence 和足够置信度。
- trace 是否记录阶段、事件和产物。

## 后续迭代切片规则

短期目标模式应优先围绕 `init` 主线选题。

每轮 milestone 必须回答：

1. 这轮改善了 `init` 用户旅程中的哪一步？
2. 用户能看到什么新的价值？
3. 它是否让成熟度诊断、差距解释或推荐行动更可信？
4. 它是否是一个可独立验收的体验切片？
5. 它是否提升了仓库理解深度、交互体验、资产质量、结果解释或验证可信度？
6. 它是否保持了渐进式协作，即用户输入发生在相关设计决策之前，并被后续步骤实际消费？

不应优先选择：

- 与 `init` 体验无直接关系的周边功能。
- 只为了内部结构漂亮、但用户看不到价值的重构。
- 只新增 schema 或测试、但没有改善 `init` 体验的技术切片。
- 大而散的多模块改造。
- 先完成全部设计再追加确认提示的伪交互。

可以选择的优先方向：

- `init` CLI 阶段化输出、颜色和进度反馈。
- 扫描后的 L0-L4 成熟度初评、下一等级差距解释和必要的维度证据展开。
- `init-summary.md` 结果解释增强。
- scan evidence 深度与 LLM-guided evidence expansion。
- guided init 的自然语言上下文补充。
- 资产生成内容的仓库特异性。
- init quality benchmark。
- 中等真实仓库的人工体验验收样本。

## 推荐演进路线

以下路线不是静态现状基线。后续每轮目标模式都必须先读取当前代码、测试和文档，实时判断这些方向中哪些仍然存在 gap，再选择一个可独立验收的纵向切片。

因此短期演进不应凭空假设新的交互载体，也不应把 Markdown 文档作为主要交互界面。演进路线应围绕当前 CLI 流程逐步打磨。

### 第一阶段：让用户看得懂

目标：用户运行 `init` 时清楚知道系统正在做什么，扫描完成后能在 CLI 中看懂系统对仓库的理解、当前成熟度等级和下一等级差距，写入后知道得到了什么、下一步该做什么。

候选 milestone：

- CLI 阶段标题、状态、颜色和进度反馈。
- 扫描结果在 CLI 中按技术栈、模块、命令、风险、不确定性分组展示。
- 成熟度初评在 CLI 中先展示 L0-L4 等级、证据、阻断项和下一步建议，再按需展开维度评分。
- 写入后的 CLI 交付摘要与 `init-summary.md` 对齐。

### 第二阶段：让判断更可信

目标：用户能在 CLI 中看到技术栈、模块、命令、风险和成熟度评分背后的证据链、置信度和不确定性，并能修正关键误判。

候选 milestone：

- stack / module / command / risk evidence 在 CLI 中分组展示。
- LLM-guided evidence expansion 根据不确定点补充读取文件。
- 用户修正内容进入 `interaction-decisions.yaml`、`context-inputs.yaml`、成熟度证据和后续生成。

### 第三阶段：让资产更定制

目标：Guide、Sensor 和 Workflow Skill 的生成必须消费前序扫描结果、用户补充、成熟度缺口和不确定性标记，生成资产明显反映目标仓库。

候选 milestone：

- Guides 注入用户确认后的真实模块、路径、命令和风险。
- Sensors 对应真实验证能力、缺口和用户补充的测试策略。
- Workflow routing policy 根据仓库风险、sensor 覆盖和用户确认的团队规则解释升级规则。
- 每个核心推荐项说明对应的成熟度维度、阻断项和预期提升。

### 第四阶段：让体验可持续

目标：已有 Harness 能持续维护，而不是每次重建；再次运行 `init` 时，用户能看到健康状态、维护建议和下一步动作。

候选 milestone：

- existing Harness maintenance triage 体验优化。
- candidate / recommendation / self-improve 与 init 入口更清晰联动。
- Runtime 过程数据只读证据进入 init 维护摘要。
